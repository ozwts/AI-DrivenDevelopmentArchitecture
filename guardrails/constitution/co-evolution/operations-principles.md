# 運用の原則

## 最高原則

**AIが安全に操作できる環境を構築する。** インフラはコードで管理し、環境により干渉を制御し、機微情報を厳格に保護する。

---

## Infrastructure as Code

**全てのインフラリソースをコードで管理する。**

- 手動操作を禁止し、再現性を保証
- 環境を数分で起動・停止・再作成できる状態を維持
- 複数AIの並行デプロイでもコンフリクトしない命名規則

---

## 環境干渉の制御

**AIの干渉範囲を環境により制限する。**

| 環境     | AIの干渉 | 目的                   |
| -------- | -------- | ---------------------- |
| ローカル | 自由     | 即座に検証             |
| 開発     | 自由     | スクラップアンドビルド |
| 検証     | 禁止     | 多層防御でアクセス不可 |
| 本番     | 禁止     | 多層防御でアクセス不可 |

### 多層防御

- ツール: 検証・本番へのアクセス手段を実装しない
- 認証・認可: 検証・本番への権限を付与しない
- 環境分離: アカウント自体を分離

### 品質ゲート

ローカル・開発環境での全テスト成功を、上位環境への移行の必須条件とする。

---

## クレデンシャル管理

**機微情報を厳格に保護する。**

### 禁止事項

- 標準出力・ファイル・ログへの出力
- コードへのハードコーディング
- バージョン管理システムへのコミット

### 許可事項

- メモリ保持のみ許可
- 環境変数またはシークレット管理ツールから取得

### 原則

- 環境分離: 開発はローカル管理、本番はシークレット管理ツール
- 最小権限: 必要なクレデンシャルのみにアクセス許可
- リソースベース認証: アクセスキーを極力発行しない

---

## 運用効率

**運用負荷とコストを最適化する。**

- マネージドサービス優先
- 自動化を重視
- 不要なリソースを作らない
- 従量課金を優先し、時間課金を避ける
