# ユーザー第一の原則

## 最高原則

**システムの都合ではなく、ユーザーの利便性を最大化する。** すべての設計判断において、ユーザーの体験を最優先とする。

---

## ユーザーの定義

**すべての利用者がユーザーである。**

| ユーザー | 利便性の観点 |
|----------|-------------|
| エンドユーザー | 迷わず目的を達成できる |
| 開発者（人間） | 理解・変更・保守が容易 |
| 開発者（AI） | 解析・生成・修正が可能 |

**本質**: システムは誰かのために存在する。その「誰か」の利便性を最大化することが、システムの存在意義である。

---

## 利便性の本質

**利便性とは、認知負荷を最小化し、目的達成を最短化することである。**

| 要素 | 原則 |
|------|------|
| 認知負荷 | 考えさせない、覚えさせない、迷わせない |
| 予測可能性 | 同じ操作は同じ結果、状態は常に把握可能 |
| 一貫性 | 同じ概念には同じ表現、例外を作らない |
| 回復可能性 | エラーから回復できる、やり直しが可能 |

---

## ユーザー体験の原則

### 時間の尊重

ユーザーの時間は最も貴重なリソース。待たせない、無駄な操作をさせない、中断しても再開できる。

### 即時のフィードバック

ユーザーは常にシステムの状態を把握できる。操作の受付、処理の進捗、結果の反映を明示する。

### 漸進的開示

必要な情報を、必要なときに、必要な分だけ。基本は直感的に、高度な機能は段階的に。

### エラーの人間性

エラーはユーザーを責めず、回復への道筋を示す。何が起きたか、次に何をすべきかを伝える。

### 自律性の保証

ユーザーが制御権を持つ。選択の自由、取り消し可能、システムの透明性。

### 包括性

すべてのユーザーが等しく利用できる。多様な能力を想定し、代替手段を提供する。

---

## 最小驚きの原則

**ユーザーの期待を裏切らない。**

- 一般的な慣習に従う
- 同じパターンは同じ動作をする
- 予想外の副作用を起こさない

---

## 優先順位

システムの都合とユーザーの利便性が衝突したとき、**ユーザーの利便性を優先する**。

| 衝突 | 優先 |
|------|------|
| パフォーマンス vs 使いやすさ | 使いやすさ（許容範囲内で） |
| 実装の簡潔さ vs 利用者の体験 | 利用者の体験 |
| 技術的な正しさ vs ユーザーの期待 | ユーザーの期待 |

**実現手段**:
- 利便性はSimpleにより実現する（`../structural-discipline/simplicity-principles.md`参照）
- 持続的な提供はスケーラビリティにより実現する（`../co-evolution/scalability-principles.md`参照）
